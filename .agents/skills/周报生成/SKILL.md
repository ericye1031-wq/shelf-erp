# 周报生成

## 功能描述

将 Markdown 格式的周报草稿内容，按参考模板格式（宋体14pt、A4页面、页脚页码），生成规范的中文 Word 文档。

## 触发方式

用户说出类似以下意图时触发：
- "帮我生成周报 Word"
- "把周报草稿转成 docx"
- "生成正式的周报文档"
- "写一份周报 Word"

## 前提条件

- `docx` npm 包已全局安装：`npm install -g docx`
- 周报参考模板：`<项目目录>/周报参考模板.docx`

## 输入

Markdown 格式的周报草稿文件，标准结构如下：

```markdown
# <姓名>——<年份>年第<周数>周周报

## 一、本周主要工作

**1. <工作模块一>**
<工作内容摘要>...

- <具体事项1>...
- <具体事项2>...

**2. <工作模块二>**
<工作内容摘要>...

## 二、下周工作计划

**1. <计划事项一>**
<具体说明>
```

### 周报内容撰写规范（重要！）

**篇幅控制**：
- 总内容控制在 **1页半以内**（A4纸），宁少勿多
- 每个子项（1./2./3.）控制在 **2句话以内**，不超过一行半

**措辞禁忌（正文中严禁出现）**：
- ❌ 版本号：如"v1.0"、"v0.1"、"v2.0"、"v3.0"
- ❌ 外来词/英文缩写："MVP"、"Demo"、"Skill"、"UI"、"API"——用中文替代或省略
- ❌ 技术型号：如"Qwen3-235B"、"Coder-30B"、"deepseek-v4"——概括写为"调整模型部署"、"部署新模型"
- ❌ 括号注释：如"（v4暂不支持PPU）"、"（任务识别→自动路由）"——括号里的技术细节全删
- ❌ 报告腔："明确核心痛点"、"输出方案"——用自然口语化表达
- ❌ 技术操作词汇："git纳管"、"截图占位符"——用"整理归档"、"补充截图"

**下周计划写法**：
- 每项用 **"推进/启动/完成/跟进" + 一句话** 概括，不超一行
- 不出现具体技术型号和英文缩写

**"其他工作"写法**：
- 合并概括写一句话，不要列技术清单

**正面示例**：
> **1. 文档学习与现状梳理**
> 系统阅读了算力资源盘点等文档，对公司当前IT基础设施和AI应用现状有了比较清晰的了解。

**反面示例**：
> ❌ "下掉Qwen3-235B/Coder-30B腾挪12卡空间"——太技术、型号太具体
> ✅ "调整模型部署，优化算力资源配置"——概括、自然

> ❌ "新增对话类请求走AI网关直接处理（参照字节Trae）"——英文+括号注释
> ✅ "优化算力调度机制，提升服务响应体验"——概括、去技术细节

## 输出

- 文件：`<姓名>-<年份>年第<周数>周周报.docx`
- 路径：与输入 markdown 同目录下
- 格式：见下方格式规范

## 格式规范（与 周报参考模板.docx 一致）

### 页面设置
- **页面**：A4（宽 11906 × 高 16838 DXA）
- **页边距**：上 1440 / 下 1440 / 左 1800 / 右 1800
- **默认字体**：宋体，14pt（sz=28）
- **行距**：360（1.5倍，lineRule="auto"）

### 标题层级
| 层级 | 字号 | 加粗 | 对齐 | 间距 |
|------|------|------|------|------|
| **主标题** | 22pt（sz=44） | 是 | 居中 | 段前 340 / 段后 330 / 行距 578 |
| **一级标题**（一、/ 二、） | 14pt（sz=28） | 是 | 左对齐 | 段前 200 / 段后 120 |
| **二级子标题**（1. / 2.） | 14pt（sz=28） | 是 | 左对齐 | 段前 120 / 段后 60 |

- **主标题格式**：`姓名——年份年第X周周报`
- **一级标题**：一、/ 二、/ 三、...
- **二级子标题**：1. 2. 3. ... 手动编号

### 正文
- **字体**：宋体，14pt（sz=28），不加粗
- **首行缩进**：480 DXA（约2字符）
- **行距**：1.5倍（360 twips）
- **段间距**：段前 60 / 段后 60

### 列表项
- 使用 `- ` 前缀，渲染为带圆点的段落
- **首行缩进**：480 DXA 后接圆点和文字
- 行距：1.5倍（360）

### 页脚
- 居中，格式：`第 PAGE 页`
- 字体：宋体，9pt（sz=18），灰色（#999999）

### 页眉
- 模板中**无页眉**，生成时不加页眉

## 操作流程

### 第一步：解析输入

1. 确定要处理的 markdown 文件路径
2. 读取文件内容（使用 Read 工具）
3. 从文件名和内容中提取：
   - 姓名（从 Markdown 标题或上下文推断）
   - 年份和周数（从 Markdown 标题中提取，如 `<年份>年第<周数>周`）

### 第二步：编写生成脚本

脚本路径：与草稿同目录，命名为 `gen_weekly_report.js`

关键格式参数（与模板对应）：

```javascript
// --- 主标题 ---
主标题：size=44, bold, center, spacing before=340, after=330, line=578

// --- 一级标题（## 一、...）---
一级标题：size=28, bold, spacing before=200, after=120

// --- 二级子标题（**1. ...**）---
二级子标题：size=28, bold, spacing before=120, after=60

// --- 正文段落 ---
正文：size=28, indent firstLine=480, spacing before=60, after=60

// --- 列表项（- 前缀）---
列表：size=28, indent left=480, spacing before=60, after=60

// --- 页脚（无页眉）---
页脚："第 PAGE 页"，sz=18, color="#999999", center

// --- 页面 ---
size: { width: 11906, height: 16838 }
margin: { top: 1440, bottom: 1440, left: 1800, right: 1800 }
spacing: { line: 360, lineRule: "auto" }
```

### 第三步：运行并验证

```bash
NODE_PATH="$(npm root -g)" node <输出目录>/gen_weekly_report.js
```

## 注意

- `NODE_PATH` 必须指定为全局 node_modules 路径
- sz 单位是 half-point：14pt=28，9pt=18，22pt=44
- A4 页面：宽 11906 × 高 16838 DXA
- 模板中无页眉，不要加页眉
- 页脚格式为 `第 PAGE 页`，灰色 #999999，9pt
- 标题不要使用 Heading 样式，用 Paragraph 直接控制字号加粗
- 参考模板路径：`<项目目录>/周报参考模板.docx`
- 实际生成脚本参考：`<输出目录>/gen_weekly_report.js`
