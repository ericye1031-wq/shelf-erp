# 业务调研报告撰写指南

## 一、需求理解与沟通（第一步，必须先做）

撰写任何业务报告前，先通过对话确认以下信息，**不要猜测**：

### 必确认清单

| # | 确认项 | 为什么重要 |
|---|---|---|
| 1 | 文档的读者是谁？（技术领导/业务领导/项目组） | 决定技术深度和业务视角的比重 |
| 2 | 目的是汇报决策还是指导执行？ | 决策型重分析结论，执行型重步骤细节 |
| 3 | 是否需要图文结合？有无截图/图表可用？ | 影响文档呈现方式和素材准备 |
| 4 | 计划要精确到时间还是弹性描述？ | 涉及外部协调/采购时，不宜精确承诺 |
| 5 | 是否需要与其他方案联动？如何关联？ | 避免错误地将独立任务写成依赖关系 |
| 6 | 技术细节写到什么程度？ | 避免过度技术化或过度简化 |
| 7 | 有无明确的「不做」范围？ | 明确排除项，防止文档范围蔓延 |

### 沟通原则

- **先问后写**：不确定的地方先问用户，不要编
- **重复确认**：对「不做」「分离」「独立」等关键词，复述确认
- **记录偏好**：用户的写作风格偏好（如：务实/正式/简洁/详细）记录到 MEMORY.md

## 二、报告结构设计

### 2.1 通用结构（适用于大多数业务调研报告）

```
一、业务背景（现状）
   → 一句话说明：现在在做什么，为什么要做这个调研

二、现状与痛点
   → 现在怎么做的？存在什么问题？
   → 用表格或流程图展示
   → 痛点要具体，能说出「谁受了什么影响」

三、方案设计（改造思路）
   → 要怎么做？分几步？
   → 每步说明：改造前 vs 改造后 的对比
   → 用截图/流程图辅助说明

四、预期效果
   → 改造后能达到什么效果？
   → 用对比表展示（改造前 vs 改造后）

五、实施计划
   → 分几步做？每步做什么？
   → 如涉及外部依赖，用弹性描述（「约X周」「取决于XX配合度」）
   → 明确关键依赖和不可控因素

六、风险与应对（可选）
   → 可能遇到什么风险？怎么应对？

七、需要决策的事项（决策型报告必备）
   → 明确列出需要领导拍板的事项
   → 每个事项说明：做什么、为什么、不做的影响
```

### 2.2 不同场景的结构调整

| 场景 | 结构调整 | 示例 |
|------|----------|------|
| **AI 赋能方案** | 突出「现有能力→痛点→改造步骤→效果对比」 | B2-差旅助手 |
| **技术可行性调研** | 增加「技术验证」章节，含截图/测试结果 | B1-机票比价 |
| **基础设施规划** | 增加「现状盘点」和「资源分配方案」，用表格展示 | 算力底座 |
| **汇报决策** | 末尾必须加「需要决策的事项」 | 所有决策型报告 |
| **指导执行** | 计划要更细，明确负责人和验收标准 | 项目执行文档 |

## 三、写作原则

### 3.1 视角选择

| 读者 | 写作视角 | 技术内容比例 |
|------|----------|-------------|
| 业务领导 | 业务视角（员工感受、效率提升、成本节约） | < 20% |
| 技术领导 | 技术+业务平衡 | 30-50% |
| 项目组/执行层 | 技术视角为主，附带业务价值说明 | 50-70% |

### 3.2 内容规范

- **痛点要具体**：不说「效率低」，说「每次操作需要 15 分钟，每天重复 10 次」
- **方案可对比**：每个改造点都要有「改造前 vs 改造后」的对比
- **计划留弹性**：涉及外部协调/供应商/采购时，不精确承诺时间节点
- **明确排除项**：明确说「不做XX」，防止范围蔓延和后续争议
- **图文结合**：有截图就嵌入，有流程就画图，不要纯文字堆砌

### 3.3 避坑清单

| 坑 | 表现 | 怎么避免 |
|---|---|---|
| **技术过度** | 满篇 API/接口/框架/配置 | 先确认读者是谁，业务领导看的不需要技术细节 |
| **计划过细** | 精确到周/天，但涉及外部依赖 | 外部依赖用弹性描述，内部可控的才精确 |
| **联动错误** | 把独立任务写成必须联动 | 确认任务间关系，独立的说清楚独立 |
| **范围蔓延** | 写着写着多了很多内容 | 明确「不做」范围，超出范围的不写 |
| **假大空** | 「提升效率」「优化体验」没有量化 | 尽量给具体数字或具体场景 |
| **重复混乱** | 时间线里写了，最后又写下一步行动 | 合并同类内容，只出现一次 |
| **编造内容** | 写自己没验证过的东西 | 不确定的标注「待确认」「需验证」「草案」 |

## 四、迭代修改规范

### 4.1 修改流程

```
用户提出修改意见
    ↓
逐条确认理解是否正确（复述一遍）
    ↓
修改文档（标注版本号变更）
    ↓
在文档末尾或独立章节写「修订说明」
    ↓
同步更新 MEMORY.md（如影响长期记忆）
```

### 4.2 修订说明格式

```markdown
> 📌 **文档状态**：vX.X
>
> **vX.X 修订说明**：
> - ❌ 删除/修改了什么（原因）
> - 🆕 新增了什么（原因）
> - ⚠️ 调整了哪些内容（原因）
```

### 4.3 版本号规则

| 版本变化 | 含义 | 示例 |
|----------|------|------|
| v1.0 → v1.1 | 小修订，内容调整 | 修改描述、补充说明 |
| v1.0 → v2.0 | 大修订，结构调整 | 重写章节、增删内容 |
| v1.0-draft | 草案，未经验证 | Skill 初版、待调试确认的方案 |

## 五、配图与素材规范

### 5.1 截图使用

- 嵌入实际产品截图（如程星途 AI 界面）增强可信度
- 截图配文字说明，不要只放图
- 多张截图并列时，用表格布局统一展示

### 5.2 流程图/架构图

- 用 SVG 或 ASCII 画图展示架构/流程
- 复杂流程分步展示，不要一张图塞太多
- 颜色统一：红色=问题/告警，绿色=完成/正常，黄色=待定/注意
